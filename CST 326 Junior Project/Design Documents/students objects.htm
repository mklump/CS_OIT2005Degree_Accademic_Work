Use case write-up

Student

 

Student logs in with key. System validates key by checking the database for validity

System acquires evaluation by adding instructor augmentations to the standard template.

Student is presented with composite evaluation from the system and the system starts a timer.

Student submits evaluation. The system validates the answers based on internal criteria and accepts or rejects the evaluation.

If the evaluation is accepted then the key used for login is destroyed in the database.

If the evaluation is NOT accepted then the student is re-presented with the evaluation form and provided with feedback as to what went wrong.

 

Nouns:

Student: Not a object, is an actor.

Key: Not a object, is a product of a object.

System: Too vague to be an object. (Booch, page 19, number 7)

Database:  Object.

Evaluation: Not a object, is a product of a object.

Instructor Augmentations: Not a object, is a product of an actor.

Standard Template: Not a object, is data used by actors and objectes

Composite Evaluation: Not a object, is data used by actors and objectes

Answers: Not a object, is data generated by an actor

Internal Criteria: Not a object, a portion of the internal logic of the system object.

Login: Object.

Timer: Object.

 

Since there are so few, I have to make some other statements:

 

Other statements:

Students login to the system using an authentication page

Students record evaluation results using an evaluation form

The system validates the evaluation form using a validation algorithm

The system saves the results using a database communicator

 

All objects: Authentication Page, Evaluation Form, Validation Algorithm, Database Communicator, Login algorithm, Timer, and Database
