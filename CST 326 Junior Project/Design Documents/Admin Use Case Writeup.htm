Admin Use case write-up

 

Admin logs in. System validates login with DB.

System presents admin with menu page. Admin can choose to modify the standard template, approve faculty augmentations, or create new evaluations.

 

Modify standard template:

System presents the admin with a page where changes can be made to the standard template. Admin makes changes and submits it to the system. If the changes are valid, then the system saves the new standard template in the database and the admin gets an ok message. If not, then they get feedback of errors.

 

Approve augmentations:

System presents admin with a list of changes pending their review. Admin makes a selection and the system displays the changes. If admin approves of the changes, then the system saves the information in the database for use in the actual evaluation. If admin denies changes then the system presents the admin with a form to send a reason they were denied, via email, to the faculty member who requested them.

System presents an ok message.

 

Create new evaluations:

System presents admin with a form to determine the evaluation name, who has access to add questions or view results for the evaluation, how many keys are needed, and open and close dates for the evaluation.

Admin submits selections to the system which validates and saves the new evaluation in the database. System generates keys in the quantity requested and saves them in the database. Admin is presented with keys for printing or can print them later, along with an OK message. Faculty member associated with the audit are emailed about its existence.

Since there are so few usable nouns describing object, I have made the following statements:

 

Administrators login to the system using an authentication page:

Administrators are then given several choices from the administrator page, where they can navigate to the 1) change evaluation form, 2) approve augmentation form, 3) create new evaluation form, or the 4) account administration form for the system accounts.

All forms require the administration to submit the form to the system for verification by a validation algorithm.

All validated changes are saved using a database communicator.
