Individual Review and Testing of Final release

Complete by: Matthew Klump

Tuesday June 8, 2004

Project: Moon Bucks - Final Release

Review another Project team's Final Release
Review the project documents and write up a brief report.  

  • do the requirements reflect the final implementation
  • does the Use Case Model reflect the final implementation
  • does the logical architecture reflect reality (the project's true functionality)
  • trace through the table that shows how the requirements map to use cases, and how use cases map to classes and methods.  Is it complete and accurate
  • etc.

Review the project/product itself and write up a brief report

  • is there an installation file or instructions on how to install the product.  If you can test the installation procedure, did it work
  • does some form of user documentation exist, whether it be a User Guide, installation guide, help files, etc.  Does it seem adequate for a new user to successfully use the product.
  • is the intended purpose of the product clear;  who are the intended customers, is it easy to use; does it do what it is stated to do

Review the project test plan and write up a brief report.  

  • were all of the tests cases successfully run and results recorded
  • are there any outstanding problems identified
  • did the test cases provide thorough coverage of the project and also common error conditions (e.g. failure to open a database, some required service, like IIS or Apache not started, etc.)

What to hand in for Individual Final Review
 Be certain to include at the top of the page your name, the project name and project deliverable you are reviewing.
Make this a constructive review document that will aid the project team to be more successful and avoid pitfalls and oversights.

  1. Write up 1-2 paragraph assessment of the reviewed project's documentation.  
  2. Complete a test report that includes a complete review of the project test plan and each test case, and the results of any testing that you were able to conduct.  Can you easily trace the test case to the use case and the defined project requirement?  What are the limitations of the testing, did the testing include error situations and hardware problems.  
  3. If possible, execute each test as defined in the test case and record the results.  You can use the form provided under "Forms" menu.
  4. What is your overall evaluation of the project.  Is it a viable product.  Is it easy to understand its purpose and use model.  Is it easy to  install, use and test.
  5. Any other comments you might have
