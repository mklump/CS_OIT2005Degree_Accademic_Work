Chapter 12: Redirection and Piping

  • Each file opened by a process assigned "file descriptor" (an integer)
  • All processes have three standard file descriptors created automatically:
    • Standard-input for input to the process (file descriptor 0)
    • Standard-output for output from the process (file descriptor 1)
    • Standard-error for errors reported by the process (file descriptor 2)
  • Abbreviated: STDIN, STDOUT, STDERR
  • Default locations:
    • STDIN comes from the keyboard
    • STDOUT and STDERR go to the screen
  • STDOUT and STDERR may be intermingled
  • All three can be "redirected" to come from or go to a file by appending the appropriate redirection character(s) and filenames at the end of the command (after all other arguments)
  • STDIN redirection
    • Two kinds: Simple and "Here-Document"
    • Simple
      • Useful for commands that usually expect input from the keyboard, and do not have a way to specify an input file
      • Uses the "less-than" character followed by the input filename (space optional)
      • Syntax: command [arguments]... < input-file
      • If input file does not exist, the shell will issue an error message
      • Examples:
        • write john < write_msg_to_john  CTRL-D to end the write command.
        • mail checkman@cs00.pcc.edu < invitation.txt
    • Here-Document
      • Useful in shell-scripts, when you do not want to create a file with the input for the command
      • Uses two "less-than" characters followed by any string you pick (no space between them)
      • Input will continue until that string appears at the beginning of a line, all by itself
      • Syntax:
        • command [arguments]... <<anystring
          one or more lines of input text...
          anystring
      • Often, anystring will be all capital letters and is often something like END_OF_TEXT, EOT, EOF or something similar
      • Examples:
        • mail -s'Report failed' checkman@cs00.pcc.edu <<END_OF_MSG
          You are receiving this automatically generated message because an error has occurred in the script that generates the weekly accounting reports. Go fix it!
          END_OF_MSG
  • STDOUT redirection:
    • If output file does not exist, it will be created
    • There are two kinds: Overwrite and Append
    • Overwrite:
      • Destroys output file if it already exists
      • Uses the "greater-than" character followed by the output filename (space optional)
      • Syntax: command [command]... > output-file
      • Examples:
        • echo "Total: " $[$quantity * $price] > order
        • sort +2 -5 cust.dat > cust_sorted.dat
    • Append:
      • Adds the output to end of the output file, if it already exists
      • Uses two "greater-than" characters followed by the output filename (space optional)
      • Syntax: command [arguments]... >> output-file
      • Examples:
        • ls -l *.eps >> eps_files
        • echo `date` >> report.txt
  • STDERR redirection:
    • Syntax is exactly the same as redirecting STDOUT, except the redirection character is preceded by the number 2 (the file-descriptor)
    • If you want to throw the error messages away, use the "null device" in place of a filename
    • Examples:
      • grep 'Smith' /common/acctg/cust.dat 2> cust_errors
      • find / -user checkman -print 2> /dev/null
  • Combining Redirection
    • Any of these redirection techniques can be combined for a single command
      • Example: sort < cust.dat >> cust_sorted.dat 2> sorting_errors
    • You can redirect STDERR to the same file as STDOUT
      • Use "&1" immediately after the STDERR redirection (no spaces)
      • You must use this syntax if you are redirecting both output and errors for overwrite to the same file
      • Example:
        • sort cust.dat > cust_sorted.dat 2> cust_sorted.dat
        • sort cust.dat > cust_sorted.dat 2>&1
  • Piping
    • Redirecting the output from one command to the input of another
    • Place a vertical bar between commands
    • Syntax: command | command
    • Examples:
      • who | grep checkman
      • sort data.txt | uniq
      • tar -c products | gzip -9 | crypt justakey | \  gzip get maximum compression
        uuencode products | mail bob@unix.com
      • example: cmd1 2>> <fileerrors> or </dev/null/> | cmd2 2>> errors | cmd3 2>>errors > out
    • Only redirects STDOUT, not STDERR
    • Examples:
      • sort data.txt | uniq 2> /dev/null
        In this example, if         data.txt does not exist, the error message from the sort command will still appear on the screen
      • sort data.txt 2> /dev/null | uniq
        In this example, if         data.txt does not exist, the error message will be thrown away
  • Tee
    • Used to redirect STDOUT to a file AND to another command (tee junction in pipeline)
    • Syntax: command | tee [-a] file... | command
    • You can specify more than one file
    • To append to files, use the -a option
    • Examples:
      • sort cust.dat | tee cust_sort.dat | uniq > cust_sort_uniq.dat

Chapter 13: Processes

  • A process is a program that is currently executing
  • Loaded into memory and is (or is about to be) performing the instructions defined in the program file.
  • Daemons
    • Any process that provides a system service--a server program
    • Usually started upon boot
    • Typically executable ends in "d" (e.g. lpd, syslogd, inetd)
    • Same thing as a "Service" in WindowsNT
  • Every process has a parent process that started it (hierarchy of processes) (try ps -eH)
  • init is the top of the hierarchy
  • Shell Command Execution
    • What happens when you type a command and press Enter
    • There are two important System Calls that a shell uses to execute commands
      • fork: Create an exact copy of the process making the call
        • Parent: The process initiating the fork
        • Child: The process created by the fork
      • exec: Replace the process making the call with the one being exec'd
    • Whenever you type a command at the shell, the following steps are followed:

1.     shell forks a child shell, and waits for the child to exit.
child is an exact copy of parent, with environment

2.     Command Execution:

a.      Binary Executable: child shell performs substitution/expansion on the command line, and uses exec to replace itself with the command to be executed

b.     Shell Script: child shell executes each of the commands in the script file (starting at step 1 above for each command)

3.     The child process exits upon completing the last instruction

  • Processes
    • All processes have attributes
    • Examples of some attributes

0.     State: {Ready|Running|Swapped|Waiting|Zombie} (see Figure 13.1, p. 320)

1.     Process ID (PID): A unique number identifying this process from all others that are currently running on the system

2.     Parent Process ID (PPID): The PID of the process that created (fork'd) this one

3.     Real User ID (UID): The UserID that actually started the process

4.     Effective User ID (EUID): The owner of the file that was executed, if SUID-bit was set on that file. Otherise, same as UID.

5.     Real Group ID (GID): The default Group ID of the UID that started the process

6.     Effective Group ID (EGID): The group owner of the file that was executed, if SGID-bit was set on that file. Otherwise, same as GID.

7.     Priority:

        • A measure of how much CPU time the process will get, relative to other processes
        • Smaller number is higher priority
        • Upper and lower bounds are different on different sytems--see p. 319
        • Users can decrease the priority (increase priority value) by using:
          • renice [adjustment] pid (for an existing process)
          • nice [-n adjustment] command (prior to execution)
        • Only superuser can increase the priority (decrease priority value).

8.     TTY: Device associated with the process

        • Device is where it will receive its input and send its output (unless redirected).
        • Undefined for processes that are not associated with a TTY device.
        • Execute a command with no TTY:
          • nohup command
          • Any output not redirected will be stored in ~/nohup.out
    • Listing running processes and their attributes: ps
      • ps has many many options and different styles of expressing them. See man pages
      • Examples:
        • ps (your processes, associated with the current TTY)
        • ps -f (your processes in a "full" format--more information)
        • ps -l (your processes in a "long" format--lots of information)
        • ps -a (processes for all users, but not including login shells)
        • ps -u username (processes for a particular user, e.g. yourself)
        • ps -e (every process)
        • ps -H (hierarchical display--indents child processes under their parents)
  • Job Control
    • Foreground
      • Commands are executed in the foreground by default
      • Shell waits for child process to exit--user cannot do anything in the meantime
    • Background
      • Commands can be executed in the background by adding an ampersand, "&", as the last argument
      • Shell does not wait for the child process to exit
    • It is wise to redirect STDOUT and STDERR for processes running in the background
    • Cancel a job running in the foreground: ^C
    • Suspend a job running in the foreground: ^Z (stops it and moves it to the background)
    • Move a (stopped or running) job to the foreground and start it: fg [jobid]
    • Start one or more stopped jobs running in the background: bg [jobid]...
    • List background jobs: jobs [jobid]...
    • jobid ::=
      • % or %+ (the most recently stopped job)
      • %- (the next most recently stopped job)
      • %N (job number N)
      • %name (job beginning with name)
      • %?name (command containing name)
    • Sequential and Parallel Execution
      • Sequential: command ; command
      • Parallel: command & command &
  • Killing processes
    • kill [-signal] {pid|jobid}... (Sends signal to pid or jobid)
      • where signal ::=
        • 15 = Software Termination (default)
        • 1 = Hangup (tells process that it has no parent)
        • 2 = Interrupt (equivalent to CTRL-C)
        • 9 = Sure kill
    • Many programs have cleanup they must do before they terminate (saving data, closing files)
    • These programs "trap" signals 15 or 1, and perform cleanup first
    • Sending signal 15 or 1 thus should initiate the cleanup and exit procedures
    • When you logout (or disconnect), your shell sends the Hangup signal (1) to all its children
    • Sometimes these signals do not cause the children to die properly
    • Zombies: Processes that are ready to exit but dont die
    • "Gremlins": Processes that are running, using up lots of resources, not doing anything useful
    • ps -u username shows all processes for username (associated with all TTYs)
    • If you see processes you own, but not connected to your TTY, you should kill them:
      • kill -1 pid (use on the child processes first)
      • ps -u (to see if it worked)
      • kill -9 pid (if it didnt work)
    • Then repeat these steps for the parent
